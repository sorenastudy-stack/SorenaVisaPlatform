## Prisma migration history is out of sync with the database (drift)
**Problem.** The local database's migration history is not consistent with the `prisma/migrations/` folder. The folder contains 60+ migrations, but the `_prisma_migrations` table records only the last few as applied — the base schema was established by `prisma db push` / a database restore, not by replaying the migration chain. As a result: (1) `prisma migrate dev` fails, because it builds a fresh shadow database by replaying every migration in order and the history can't replay cleanly (it hits an `ALTER`/`ADD VALUE` against a table a later migration is expected to have created); and (2) `prisma migrate diff` between the live DB and `schema.prisma` reports significant unrelated drift (it would, for example, `DROP TABLE country_migration_log`, drop indexes, and alter several tables).

**Risk.** This is a latent danger for production deploys. A future `prisma migrate:deploy` (or `migrate dev`) run against an environment whose history is in this state could fail outright, or — worse — attempt destructive changes (dropping tables/columns) while "reconciling" the perceived drift. `db push` is equally unsafe here because it would sweep up all of that unrelated drift rather than just the intended change. Until the history is reconciled, neither `migrate dev`, `migrate deploy`, nor `db push` can be trusted to make a clean, intention-limited change.

**Recommended fix (do NOT do this as part of a feature PR).** Reconcile the migration history so local and production share one clean, consistent chain before relying on `migrate dev` again. The usual approach is to baseline/squash: snapshot the current production schema as a single fresh "init" migration, reset the `migrations/` folder to that baseline, and mark it applied (`migrate resolve --applied`) on every environment so the recorded history matches the actual schema. This is a dedicated infrastructure task with its own review — it must be planned against the real production schema, not done casually.

**How additive migrations should be applied safely in the meantime.** Until the baseline is fixed, do NOT run `migrate dev` / `db push` for schema changes. Instead, follow the pattern used for the booking foundation (`20260629000000_pr_booking_1_foundation`): (1) edit `schema.prisma` with purely additive changes (new tables, nullable columns, new enum values, new indexes — never drop/rename/retype, never add NOT NULL to existing columns); (2) hand-author the `migration.sql` with idempotent guards (`IF NOT EXISTS`, `ADD VALUE IF NOT EXISTS`, `DO $$ ... pg_constraint check ... $$`); (3) apply it to the target DB with `prisma db execute --file ...` (which does not touch migration history); (4) record it with `prisma migrate resolve --applied <name>` so the folder and `_prisma_migrations` stay aligned; (5) `prisma generate`. Verify object-by-object afterwards (the booking migration was verified to add only its own table/columns/indexes and to leave the unrelated drift untouched). This keeps each change intention-limited and non-destructive despite the broken history.
